## Change Data Capture (CDC) Architecture

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ PHASE 2: CHANGE CAPTURE & CONTINUOUS SYNC (overlaps weeks 2-11)           │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Oracle REDO Logs (archival)                                                │
│  ├─ Generated by all DML on source tables                                   │
│  ├─ Stored in Oracle archive location                                       │
│  └─ Available for LogMiner query (7+ days retention)                       │
│         │                                                                    │
│         ▼                                                                    │
│  LogMiner Repository                                                        │
│  ├─ Redo log dictionary loaded per session                                 │
│  ├─ V$LOGMNR_CONTENTS queried for DML rows                                │
│  └─ Filtered by start SCN (snapshot point) and table list                 │
│         │                                                                    │
│         ▼                                                                    │
│  Python CDC Extractor (scripts/oracle_logminer_cdc.py)                     │
│  ├─ Maintains connection to Oracle                                         │
│  ├─ Polls LogMiner for new changes (batch mode, 1000-10000 rows)          │
│  ├─ Extracts OPERATION, UNDO_SQL, REDO_SQL, row_id context               │
│  ├─ Converts Oracle types → PostgreSQL types                               │
│  ├─ Handles sequences, LOBs, special characters                            │
│  ├─ Tracks current high-water-mark SCN (for resume on restart)             │
│  ├─ Output: JSON Lines (\n-delimited JSON) to CDC queue                    │
│  └─ Parallel instances (one per table group or schema partition)           │
│         │                                                                    │
│         ├─ JSON Queue: /var/lib/cdc/changes.jsonl                           │
│         │  ├─ INSERT: {"operation":"INSERT","table":"customers",           │
│         │  │           "data":{"id":1,"name":"John","created":...}}       │
│         │  ├─ UPDATE: {"operation":"UPDATE","table":"customers",           │
│         │  │           "pk":{"id":1},"before":{...},"after":{...}}        │
│         │  └─ DELETE: {"operation":"DELETE","table":"customers",          │
│         │             "pk":{"id":1}}                                        │
│         │                                                                    │
│         ▼                                                                    │
│  CDC Replay Engine (scripts/cdc_replay_postgresql.py)                      │
│  ├─ Consumes JSON queue (streaming or batch)                               │
│  ├─ Parses JSON and route by table                                         │
│  ├─ Prepare SQL (INSERT, UPDATE, DELETE for PostgreSQL)                   │
│  ├─ Handle idempotency (ON CONFLICT, IF EXISTS logic)                     │
│  ├─ Batch apply (1000-5000 rows per transaction)                          │
│  ├─ Track applied changes (high-water-mark checkpoint table)               │
│  ├─ Log replay progress (public.cdc_checkpoint table)                      │
│  └─ Exponential backoff on connection loss                                 │
│         │                                                                    │
│         ▼                                                                    │
│  PostgreSQL Target                                                          │
│  ├─ Apply UPDATEs/DELETEs via PK lookup (fast, indexed)                   │
│  ├─ Apply INSERTs directly                                                 │
│  ├─ Sequences updated by CDC (SETVAL after INSERT/UPDATE)                  │
│  └─ CDC lag monitored (target < 5 min behind Oracle source)               │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
  Weeks 2-11: Continuous sync, CDC lag < 5 minutes typical
```

## Deployment Topology

### Infrastructure Requirements

**Oracle Source**
- Version: 12c, 18c, 19c, or 21c
- Edition: EE or SE (both supported)
- Database Size: 120+ TB
- ARCHIVELOG mode: Required (for LogMiner)
- Supplemental Logging: Enabled on migrated tables
- Network: 1+ Gbps to Data Box staging area

**PostgreSQL Target**
- Version: 14, 15, or 16
- Compute: Min. 32 vCPU, 256 GB RAM (Azure Database for PostgreSQL Flexible Server: General Purpose SKU)
- Storage: 150+ TB (SSD, provisioned IOPS for fast indexing)
- Extensions: pgcrypto, pg_trgm (for string operations)
- max_connections: 500+
- shared_buffers: 65 GB (25% of RAM)
- effective_cache_size: 200 GB (75% of RAM)

**Azure Infrastructure**
- Data Box devices: 2-4 × 1 TB units (for parallel shipment)
- Data Lake Storage Gen2: Container with Hierarchical Namespace enabled
- Data Factory: Standard pricing tier
- Virtual Network: 1+ Gbps ExpressRoute or VPN to Oracle network

**Extraction Host** (Oracle CSV export machine)
- CPU: 16+ cores
- RAM: 128 GB
- Storage: 200 GB local (for buffering)
- Network: Direct connection to Oracle (preferably same LAN)
- Tools: Oracle client, expdp/sqlplus, Python 3.9+

**CDC Extractor Host** (LogMiner polling + PostgreSQL writer)
- CPU: 4-8 cores
- RAM: 32 GB
- Storage: 50 GB (for CDC queue)
- Network: Dual connection (Oracle + PostgreSQL)
- Tools: Python 3.9+, cx_Oracle, psycopg2

## Data Flow Sequences

### Sequence 1: Initial Bulk Load
```
1. Oracle audit → identify 1200 tables, 120 TB total
2. Schema extraction: sqlplus → dbms_metadata → DDL scripts
3. DDL conversion: ora2pg + custom Python → PostgreSQL CREATE statements
4. Create empty target tables in PostgreSQL (no FKs yet)
5. Parallel expdp: 8 parallel dumps, one per 15 GB partition
6. Generate CSVs: 800 CSV files, each ~150 MB
7. Stage to Data Box device
8. Ship to Azure; ADLS ingestion (automatic)
9. CSV → Parquet conversion: 8 parallel jobs, ~100 files at a time
10. ADF pipeline: 50 concurrent COPY writers, 1200 tables load in ~4 days
11. Post-load validation: row counts, sample checksums
12. Enable FKs and build indexes (~1 day)
```

### Sequence 2: CDC Capture
```
1. Record snapshot SCN: 123456789 (point-in-time for initial load)
2. Enable supplemental logging on all migrated tables
3. Archive redo logs (ARCHIVELOG mode running)
4. Start CDC extractor: oracle_logminer_cdc.py --start-scn 123456789
5. LogMiner session created, redo logs added to logminer
6. Poll V$LOGMNR_CONTENTS for DML rows (committed only)
7. For each row: extract table, pk, operation, undo_sql, redo_sql
8. Convert types: Oracle NUMBER → numeric, DATE → TIMESTAMP, etc.
9. Output JSON: {"operation":"UPDATE","table":"customers", ...}
10. CDC replay: consume JSON, apply to PostgreSQL target
11. Track checkpoint: high_water_mark = current SCN
12. Repeat: every 5 seconds (tunable polling interval)
```

### Sequence 3: Cutoff & Go-Live
```
1. Quiesce source: set Oracle tables to READ ONLY mode
2. Final CDC pass: pull any last changes (should be minimal)
3. Wait for replay: CDC lag = 0 (all changes applied)
4. Final reconciliation: row counts, PKs, checksums
5. Sign-off: all checks passed
6. Switch application: update connection string → PostgreSQL
7. Smoke test: 20-30 application queries, verify results
8. Go-live confirmed!
9. Keep Oracle READ ONLY for 72 hours (rollback window)
10. Post-cutoff: monitor PostgreSQL, check query performance
```

## Failure Recovery & Rollback

### Scenario: Initial Load Fails at Table N/1200
```
1. Identify failed table from ADF pipeline log
2. Investigate: data type issue? Network timeout? PG constraint?
3. Fix root cause (e.g., adjust PostgreSQL WORK_MEM, rebuild index)
4. Resume: ADF pipeline restarts from table N (idempotent COPY with conflict handling)
5. Continue until all 1200 tables loaded
6. Re-validate row counts and checksums
```

### Scenario: CDC Extractor Crashes
```
1. Detect: monitoring alerts (no checkpoint update for 10 min)
2. Check logs: ora_logminer_cdc.log for error
3. Restart: python scripts/oracle_logminer_cdc.py --resume (reads last SCN from checkpoint)
4. Resume change capture from last applied SCN (no data loss)
5. Monitor: verify replaying backlog without lag spike
```

### Scenario: PostgreSQL Target Disk Full
```
1. Alert: pg_monitor detects 95% disk capacity
2. Action: expand PGDATA volume (Azure VM disk resize)
3. Resume: once disk > 50% free, CDC replay continues
4. Monitor: CDC lag should return to normal <5 min
```

### Rollback to Oracle (if cutoff issues detected)
```
1. Trigger: post-cutoff issue detected (data mismatch, performance)
2. Switch: update application connection string → Oracle (pre-staged)
3. Scope: re-enable Oracle for writes (EXIT READ ONLY mode)
4. Investigate: analyze discrepancy in PostgreSQL (post-incident)
5. Re-attempt: fix issue, restart cutoff process
```

## Monitoring & Observability

### KPIs Tracked

| KPI | Target | Alert Threshold |
|-----|--------|-----------------|
| **Initial Load Progress** | 120 TB in 4 days | >5 days → escalate |
| **ADF Pipeline Success Rate** | 100% | <98% → investigate |
| **CDC Lag** | <5 minutes | >15 min → investigate |
| **Reconciliation Row Count** | 100% match | >0.1% mismatch → blocker |
| **Checksum Validation** | ≥99.9% match | <99.9% → investigate |
| **PostgreSQL CPU** | <70% | >85% → tune/scale |
| **PostgreSQL Disk I/O** | <50% util | >70% → add IOPS |
| **Query Performance** | Within ±20% of Oracle | >20% slower → tune index |

### Logs & Artifacts
- `migration.log` – main execution log (all phases)
- `cdc_progress.log` – CDC checkpoint tracking (SCN, timestamp, rows applied)
- `adf_pipeline.log` – Azure Data Factory run history
- `reconciliation_report.txt` – final validation summary
- `business_validation_report.txt` – application query results
- `performance_baseline_report.txt` – query timing comparison

## Version Control & Change Management

All migration artifacts are version-controlled:
- **DDL scripts**: version in git (track schema changes)
- **Python scripts**: pin dependencies (requirements.txt)
- **Config files**: templated, env vars for secrets
- **SQL validation scripts**: part of project repo

Pre-cutoff review:
1. Code review of custom CDC extraction logic
2. SQL review of reconciliation queries
3. Test run in non-prod environment (if time permits)
4. Approve change request (change board sign-off)
5. Cutoff window approved by stakeholders
